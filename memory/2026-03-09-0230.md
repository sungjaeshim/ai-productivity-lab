# 1) 문제 정의

지금 구조의 핵심 문제는 이겁니다.

**현재**
- 어떤 작업은 Telegram DM으로 감
- 어떤 작업은 Discord 채널로 감
- 어떤 건 cron마다 직접 target 하드코딩
- 어떤 건 topic 개념 없이 “그냥 보내기” 위주

**결과**
- 규칙이 코드/프롬프트/크론에 흩어짐
- 새 작업 추가 때마다 “어디 보내지?”를 다시 판단해야 함
- 토픽 운영이 커질수록 누락/중복/오발송 위험 증가

즉, 지금 필요한 건 단순 매핑이 아니라:

> **출력의 의미를 먼저 표준화하고, 그 의미를 목적지 토픽으로 연결하는 규칙 시스템**

# 2) 설계 원칙

## P1. 입력 라우팅과 출력 라우팅을 분리
- 입력 바인딩: 어떤 대화가 어떤 agent/session으로 들어오나
- 출력 라우팅: 생성된 결과를 어디로 보내나

이 둘을 섞으면 꼬입니다.

예:
- 사용자는 topic 37에서 요청
- 결과는 topic 78로 가야 할 수 있음
- 따라서 “입력 토픽 = 출력 토픽”으로 고정하면 안 됨

## P2. `agent`가 아니라 `message class`가 주체
같은 `main` agent라도 출력 성격은 다릅니다.

예:
- 운영 알림
- 시장 브리핑
- 승인 요청
- 로그/디버그
- 개인 리마인더
- HQ 요약

이걸 agent별로 묶으면 장기적으로 실패합니다.

## P3. 기본값보다 예외 규칙이 적어야 함
설계는 “복잡한 예외를 많이 적는 것”이 아니라:
- 기본 경로 1개
- 중요 예외 몇 개
- 나머지는 fallback

이어야 안정적입니다.

## P4. 실패 시 조용히 증발하면 안 됨
라우팅 실패는 최소한 아래 중 하나여야 합니다.
- fallback destination으로 전송
- 전송 보류 + audit log
- 운영 경고 1회

## P5. 먼저 dry-run 가능해야 함
실제 전송 전에:
- “이 메시지는 원래 어디로 갔어야 했나”
를 기록할 수 있어야 합니다.

# 3) 제안 아키텍처: 5층 구조

저는 아래 5층으로 가는 걸 추천합니다.

## Layer 1. Source Binding
입력 출처를 agent/session에 연결

예:
- Telegram group topic 37 → main
- Telegram group topic 78 → main
- Discord channel X → main
- Discord thread Y → acp session

이건 이미 일부 있음.

## Layer 2. Message Classification
출력을 **의미 클래스**로 분류

예:
- `approval_request`
- `ops_alert`
- `hq_summary`
- `market_brief`
- `market_signal`
- `daily_report`
- `weekly_report`
- `system_maintenance`
- `task_completion`
- `debug_log`
- `user_reply`
- `reminder`
- `cross_channel_mirror`

이 레이어가 핵심입니다.

## Layer 3. Destination Policy
각 class가 가야 할 목적지 정의

예:
- `ops_alert` → Telegram urgent topic
- `hq_summary` → Discord #jarvis-hq
- `market_signal` → Telegram signal topic + Discord briefing-log
- `debug_log` → Discord logs thread only
- `approval_request` → Telegram approval topic

## Layer 4. Delivery Guard
전송 전 최종 검증

예:
- group에 보내도 되는가
- 이 class는 DM 전용인가
- 길이 제한 넘는가
- 동일 본문 중복인가
- 고정 포맷 필요한가

## Layer 5. Audit / Replay
결정 로그 저장

예:
- source=session:cron:abc
- class=market_signal
- chosen_dest=telegram:-100...:topic:78
- fallback_used=no
- delivery=success

이게 있어야 나중에 “왜 저기로 갔지?”를 추적할 수 있습니다.

# 4) 핵심 설계: Message Class 체계

이 설계의 성패는 class 설계에 달려 있습니다.
처음부터 너무 세분화하면 망합니다.
그래서 **V1은 9~12개 정도**가 적당합니다.

## 추천 V1 classes

### A. `user_reply`
사용자의 현재 대화에 대한 직접 응답
- 원칙: **현재 대화 컨텍스트에 reply**
- 예: 지금 제가 드리는 이 답변

### B. `approval_request`
의사결정/승인 필요
- 예: restart, config patch, deploy, external send 전 승인 요청
- 기본 목적지: 사용자가 보고 즉시 판단 가능한 토픽

### C. `ops_alert`
운영 이상/실패/주의
- 예: 크론 실패, 라우팅 실패, API 인증 실패, 디스크 부족
- 기본 목적지: 긴급 대응용 토픽

### D. `hq_summary`
운영 전체를 보는 요약
- 예: morning briefing, evening summary, daily HQ card
- 기본 목적지: HQ 토픽/채널

### E. `market_signal`
즉시성이 높은 시그널
- 예: MACD cross, VIX alert, 급변 감지
- 기본 목적지: signal 전용 토픽
- 조건부로 mirror 가능

### F. `market_brief`
분석/브리핑
- 예: 프리마켓 브리핑, 장마감 브리핑, 주간 시장 요약
- 기본 목적지: briefing/log 토픽

### G. `task_completion`
작업 완료/증빙 보고
- 예: subagent 완료 보고, 구현 완료, 검증 완료
- 기본 목적지: 요청이 발생한 토픽 또는 프로젝트 토픽

### H. `debug_log`
운영자용 로그/디버깅
- 예: stack trace, verbose diagnostics, trace bundle
- 기본 목적지: logs thread/channel only
- Telegram group 기본 금지

시간 기반 리마인더/넛지

### J. `maintenance_report`
백업/인덱스/업데이트/정리 결과
- 기본 목적지: ops 또는 HQ

원본이 이미 있는 메시지의 미러 전송

# 6) 라우팅 결정 규칙

라우팅 엔진은 아래 순서로 판단하면 됩니다.

## Rule Order

### R1. Direct reply 우선
- 현재 사용자 대화에 대한 답변이면 `user_reply`
- 무조건 현재 컨텍스트 우선
- 단, 시스템이 생성한 background output이면 제외

### R2. Explicit override 우선
작업이 명시적으로 지정한 경우
- 예: “이건 Discord HQ로 보내라”
- 예: cron metadata에 `routeClass=market_signal`
- 명시 규칙 > 일반 규칙

### R3. Message class 기반 기본 경로
- `market_signal` → `tg.signal`, `dc.briefing`
- `ops_alert` → `tg.ops`
- `hq_summary` → `dc.hq`

### R4. Session affinity 보조
- 특정 프로젝트/토픽에서 발생한 작업이면 그 affinity를 참고
- 예: growth 프로젝트 작업 완료는 `dc.project.growth`

단, affinity는 class보다 아래여야 합니다.

### R5. Fallback
- class 미분류 → `tg.approval` 또는 `dc.logs`
- 전송 실패 → secondary destination
- secondary도 실패 → audit only + ops alert 1회

# 7) 가장 중요한 설계 결정: “입력 토픽 유지”를 기본값으로 둘 것인가?

제 답은:

> **직접 대화 응답만 yes, 시스템 출력은 no**

## A. direct conversational reply
- 사용자가 묻고 내가 답하는 것
- 원칙: 현재 topic 유지
- 이유: 대화 일관성

## B. autonomous/system output
- cron, batch, subagent completion, monitoring alert
- 원칙: **의미 기반 목적지**
- 이유: 요청 발생 위치와 결과 소비 위치가 다를 수 있음

예:
- topic 37에서 설정 논의
- 실제 시장 경보는 signal topic 78로 가야 함
- 따라서 system output은 current topic inherit 하면 안 됨

이게 이번 설계의 핵심 분기입니다.

- `hq_summary` → Discord 원본, Telegram은 요약만 선택적
- `ops_alert` → Telegram 원본, Discord는 심각도 high일 때만 mirror

# 10) 하드코딩을 어떻게 줄일까

현재 cron/job 프롬프트에 이런 게 많습니다.
- `message send channel=telegram target=62403941`
- `discord target=147...`

이 구조는 빨리 쓸 수는 있지만, 규칙화와는 반대 방향입니다.

## 권장 방향
각 job이 직접 target을 모르지 않게 하지 말고,
최소한 아래 둘 중 하나를 쓰게 해야 합니다.

### 옵션 A. class 기반
job은 이렇게만 선언:
- `routeClass=market_signal`
- `routeClass=maintenance_report`

라우터가 실제 목적지 결정

### 옵션 B. destination alias 기반
job은 이렇게 선언:
- `routeTo=tg.signal`
- `mirrorTo=dc.briefing`

숫자 ID는 중앙 설정이 해결

## 제 추천
**V1은 B, V2는 A+B 혼합**
왜냐면:
- 바로 적용 쉽고
- 사람도 읽기 쉬움
- 나중에 class 자동화 붙이기 좋음

# 11) fallback 설계

fallback은 반드시 명시해야 합니다.

## 기본 fallback
- Telegram class 미매핑 → `tg.approval`
- Discord class 미매핑 → `dc.logs`

## 전송 실패 fallback
- primary 실패 → secondary 1회
- secondary 실패 → `ops_alert` 1회 + audit log

## 금지 fallback
- `debug_log`를 Telegram group으로 흘리는 fallback 금지
- `approval_request`를 logs only로 보내는 fallback 금지
- `market_signal`를 무작정 current topic으로 보내는 fallback 금지

# 12) dedupe / noise 제어

토픽 운영이 커지면 중복이 진짜 아픕니다.

## dedupe key 제안
- `class + source_job + normalized_title + time_bucket`

예:
- `market_signal|nq_macd|golden_cross|2026-03-09T10:30`
- 동일 키는 Telegram/Discord 각 채널별 1회만 허용

## noise rule
- `ops_alert`는 동일 오류 반복 시 suppress
- `hq_summary`는 고정 시간 외 추가 전송 금지
- `task_completion`은 동일 task id당 완료 보고 1회만

# 14) 점진 도입안

big bang 하면 망합니다.
그래서 3단계가 좋습니다.

## Phase 1. Alias + 정책표만 도입
- 목적지 alias 정의
- class 표 정의
- 기존 job은 숫자 target 유지
- 대신 신규/수정 job부터 alias 사용

목표:
- 기준면 고정

## Phase 2. Dry-run router
- 실제 전송은 기존 방식 유지
- 동시에 “이 메시지는 원래 어디로 가야 했는지” 로그만 남김

목표:
- 오분류 검증

## Phase 3. 실제 적용
- 시장 시그널, HQ summary, ops alert 3개 class부터 라우터 적용
- 나머지는 점진 이전

목표:
- 리스크 작은 고가치부터 전환
