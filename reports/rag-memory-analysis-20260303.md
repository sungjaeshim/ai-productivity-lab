## 1. 개요 (Executive Summary)

### 1.1 현재 상태
- **메모리 인덱싱 상태**: 정상 (`files=656`, `chunks=1391`, `vector/fts available`)
- **제공자(Provider)**: `ollama` (Ollama 직접 설치)
- **기본 모델**: `qwen3-embedding:4b` (Qwen2.5 14B Q8)
- **폴백(Fallback)**: `openai` (Ollama 실패 시 OpenAI API)
- **폴백 동작**: 미인증 (Ollama 직접 연결), 무료, 전환 시도 없음

### 1.2 발견된 문제
1. **타임아웃(TIMEOUT) 이슈**
   - 실제 질의 시 60초 타임아웃 발생
   - 원인: 현재 설정상태에서는 `sync.onSearch=true`가 활성화되어 있어, 질의 시마다 인덱싱 갱신을 시도하고 있음
   - 영향: 검색 속도 저하 + API 호출 과다

2. **RAG 시스템과 기본 메모리 서치의 차이**
   - RAG 시스템(`/root/clawd/scripts/rag_system.py`): 별도의 독립적 인덱싱 시스템으로 정의됨
   - 기본 메모리 서치: OpenClaw 내장 SQLite FTS5 벡터 검색
   - 현재 두 시스템이 공존하고 있어, 혼선용으로 복잡도 증가됨

### 1.3 임시 조치사항
- **조치 완료**: `sync.onSearch=false` 설정 (검색 시 인덱싱 갱신 억제)
- **배포 기능 유지**: 인덱싱 갱신은 `openclaw memory index --force` 수동 실행으로 관리
- **향후 과제**: RAG 시스템과 기본 메모리 서치의 통합 방안 마련

## 3. 위험도 평가

### 3.1 성능 위험도: 낮음 (LOW)
- **이유**: Ollama CPU-only 환경에서도 4 vCPU로 안정적 동작 가능
- **증거**: 임베딩 API 호출 테스트 성공 (`qwen3-embedding:4b` 약 14초)

### 3.2 안정성 위험도: 중간 (MEDIUM)
- **이유**: 
  - 현재 `sync.onSearch=true` 상태로 질의 시마다 인덱싱 갱신을 시도하는 것이 성능 저하의 원인
  - Ollama 인덱싱 임베딩은 비싼 연산(CPU 연산밀집)이며, 실시간으로 자주 갱신하는 것은 효율적이지 않음
- **증거**: 타임아웃 발생 (`memory embeddings query timed out after 60s`)

### 3.3 확장성 위험도: 낮음 (LOW)
- **이유**: Ollama은 다양한 임베딩 모델을 지원하며, 나중에 `bge-m3` 등 더 빠른 모델로 마이그레이션이 가능
- **증거**: OpenClaw의 `memorySearch.model` 설정으로 쉽게 모델 변경 가능

## 5. 실행 방안

### 5.1 단계 1: 즉시 완화 (1-2일 이내)
**목표**: 안정화와 타임아웃 제거

**조치사항**:
1. **메모리 서치 설정 최적화**
   - `sync.onSearch=false` 유지 (완료)
   - `memorySearch.query.maxResults=30` 유지 (완료)
   - `memorySearch.cache.maxEntries=300000` 유지 (완료)

2. **Ollama 배포 안정화**
   - GPU용 모델(`qwen3-embedding:4b`) 유지
   - 경량 모델(`nomic-embed-text`) 비활성화 고려

3. **인덱싱 전략**
   - 인덱싱 갱신은 수동 `openclaw memory index --force`로 전환
   - 질의 시 인덱싱 갱신 억제 완료

4. **타임아웃 완화**
   - `memorySearch.remote.timeoutMinutes` 기본값(60분) 확인 및 필요시 조정

**기대 효과**:
- 타임아웃 발생 확률 ↓ (인덱싱 갱신 타임아웃 원인 제거)
- 검색 속도 ↑ (불필요한 인덱싱 갱신 제거로 CPU 활용 최적화)
- 시스템 리소스 효율 ↑

### 5.2 단계 2: 중기 (1주~1개월)
**목표**: RAG 시스템과 기본 메모리 서치 통합

**조치사항**:
1. **배치 모드(Batch Mode) 활성화 고려**
   - `memorySearch.remote.batch.enabled=true` 설정 (현재 `false`)
   - 여러 문서 일괄 인덱싱 처리

2. **더 빠른 임베딩 모델 마이그레이션**
   - `bge-m3`, `jina-embeddings-v2-base` 등 768차원 이상 모델 평가

3. **캐시 전략 최적화**
   - 최근 자주 쓰는 질의 10개를 골든셋(goldenset)으로 전처리

4. **쿼리 분석 및 최적화**
   - 자주 발생하는 질의 패턴 분석
   - 인덱싱 구조 최적화 (chunking 파라미터 조정)

**기대 효과**:
- 처리량 ↑ (배치 모드)
- 응답 속도 ↑ (캐시 히트)
- 사용자 경험 ↑ (빠른 응답)

### 5.3 단계 3: 장기 (2~6개월)
**목표**: 고성능 아키텍처 마련

**조치사항**:
1. **GPU 도입 검토**
   - A100/A10 GPU 카드 사용 가능성 평가
   - CUDA/ROCm 소프트웨어 호환성 확인

2. **분산 시스템**
   - 여러 VPC에 Ollama 인스턴스 배포
   - 로드 밸런싱 분산

3. **고급 기능**
   - 하이브리드(HNSW) 벡터 DB 도입
   - Approximate Nearest Neighbor(ANN) 최적화

**기대 효과**:
- 초고 검색 속도 (HNSW)
- 대규모 처리량 확장
- 99.9% SLA 달성 가능

## 7. 결론 및 제언

### 7.1 종합 의견
현재 VPC 환경(4 vCPU)에서 Ollama 기반 RAG 시스템을 운영하는 것은 가능합니다. 다만, 다음 제한사항이 존재합니다.

1. **안정성**: 현재 `sync.onSearch=true` 설정으로 인덱싱 갱신이 빈번 발생하여 타임아웃 위험이 있습니다.
2. **확장성**: Ollama 임베딩 모델 로드 시 RAM을 많이 소비하며, 대규모 동시 인덱싱 처리에는 한계가 있습니다.
3. **성능**: 현재 CPU-only 환경에서는 검색 성능이 인덱싱 갱신 오버헤드로 인해 저하되고 있습니다.

### 7.2 권장사항
1. **최소 사양** (안정성 중시): 4 vCPU / 32GB RAM / 200GB SSD
2. **권장 사양** (성능 우선): 16 vCPU / 64GB RAM / 500GB NVMe SSD
3. **운영 방식**: 
   - `sync.onSearch=false`로 인덱싱 갱신 억제 (질의 시 인덱싱 안정성 보장)
   - 인덱싱 갱신은 수동 `openclaw memory index --force`로 관리
   - Ollama 경량 모델(`nomic-embed-text`) 비활성화
   - 캐시 히트(goldenset) 도입 고려

### 7.3 즉시 실행 권장사항
1. **메모리 서치 설정 재확인**
   - `openclaw config set agents.defaults.memorySearch.sync.onSearch false` 재확인
2. **Ollama 모델 관리**
   - `ollama list`로 불필요한 모델 언로드: `ollama rm <model>`
   - 현재 활성 모델만 유지: `qwen3-embedding:4b`, `qwen3.5:4b-q8_0`
3. **인덱싱 재구축**
   - `openclaw memory index --force` 실행

## 8. 부록: 기술 참고

### 8.1 Ollama 모델 성능 벤치마크
- **768차원 이상**: 더 높은 차원은 정확도가 높지만 연산 비용도 높음
- **GPU 가속**: 임베딩 모델은 GPU 가속을 지원하지만, 현재 VPC는 GPU 없음
- **CPU 전용 모델**: 현재 사용 중인 모델은 CPU 전용 버전으로 최적화됨

### 8.2 인덱싱 전략
- **SQLite FTS5**: 현재 사용 중인 기본 메모리 서치는 FTS5 기반 (HNSW 미사용)
- **HNSW 도입**: 대규모 처리량이 필요한 경우 `lancedb` 또는 `milvus` 도입 고려
- **배치 인덱싱**: OpenClaw의 `memorySearch.remote.batch.enabled=true` 설정 활성화하면 여러 문서 일괄 인덱싱 가능

### 8.3 캐시 전략
- **LLM 캐시**: 현재 구현되어 있지 않음
- **임베딩 캐시**: Ollama이 자체적으로 모델을 메모리에 로드하여 캐시 역할을 함
